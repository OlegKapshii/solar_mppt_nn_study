# Класичний MPPT-алгоритм Perturb & Observe (P&O)

## Принцип роботи

P&O — найпростіший і найпоширеніший алгоритм відстеження точки максимальної потужності (MPP). Не потребує знання моделі панелі, працює лише за вимірюваннями струму та напруги на виході.

**Ідея:**
1. Слегка змінюємо (perturb) робочу напругу панелі на крок ΔV.
2. Спостерігаємо (observe) — як це вплинуло на потужність.
3. Якщо потужність зросла — продовжуємо рухатись у тому ж напрямку.
4. Якщо потужність зменшилась — розвертаємось.

## Математика

Позначимо:
- V(k), P(k) — напруга і потужність на поточному кроці;
- V(k-1), P(k-1) — на попередньому;
- ΔV = V(k) - V(k-1), ΔP = P(k) - P(k-1).

Закон керування — таблиця прийняття рішень:

| ΔP    | ΔV    | Дія               |
|-------|-------|-------------------|
| > 0   | > 0   | V ← V + ΔV_step   |
| > 0   | < 0   | V ← V - ΔV_step   |
| < 0   | > 0   | V ← V - ΔV_step   |
| < 0   | < 0   | V ← V + ΔV_step   |

Тут ΔV_step — параметр алгоритму (типово 0.1-1% від V_oc).

## Блок-схема

```
       ┌───────────────────────┐
       │ Початок (V₀ = 0.7·Voc)│
       └───────────┬───────────┘
                   ▼
   ┌──────────────────────────────────┐
   │ Виміряти V(k), I(k), P=V·I       │
   └───────────────┬──────────────────┘
                   ▼
       ┌──────────────────────┐
       │ ΔP = P(k) - P(k-1)   │
       │ ΔV = V(k) - V(k-1)   │
       └───────────┬──────────┘
                   ▼
        ┌─────────────────────┐
        │ ΔP·ΔV > 0 ?         │
        └────┬───────────┬────┘
        TAK │           │ НІ
            ▼           ▼
   ┌─────────────┐  ┌────────────────┐
   │ V += ΔV_step│  │ V −= ΔV_step  │
   └──────┬──────┘  └───────┬────────┘
          │                 │
          └────────┬────────┘
                   ▼
       ┌──────────────────────┐
       │ Зберегти V(k), P(k)  │
       │ Чекати наступного    │
       │     такту            │
       └──────────────────────┘
```

## Реалізація — `trackers/mppt_po.m`

Ключові рядки:

```matlab
dP = P_prev - state.P_last;
dV = V_prev - state.V_last;

if dP > 0
    if dV >= 0,  V_cmd = V_prev + step;
    else,        V_cmd = V_prev - step;
    end
else
    if dV >= 0,  V_cmd = V_prev - step;
    else,        V_cmd = V_prev + step;
    end
end
```

Стан зберігається у struct `state` з полями `V_last`, `P_last`, `step_count`.

## Параметри

- `cfg.po.V_step` — крок збурення (за замовч. 1.0 В на рівні масиву 329 В).
- `cfg.po.V_init_frac` — початкова напруга як частка від Voc масиву (0.7 = 70% від холостого ходу).
- `cfg.po.update_every_s` — частота оновлення (1 с).

## Переваги

1. **Простота реалізації** — кілька умовних операторів, мінімум обчислень.
2. **Не потребує знання моделі панелі** — працює «всліпу», тільки за P і V.
3. **Незалежність від вимірювання G, T** — не потребує датчиків освітленості.
4. **Робастність** — навіть при погано налаштованих параметрах знаходить наближення MPP.
5. **Низька обчислювальна складність** — придатний для дешевих 8-біт мікроконтролерів.

## Недоліки

1. **Steady-state oscillation** — у стаціонарному режимі алгоритм постійно «гойдається» біля MPP з амплітудою ±ΔV_step. Втрати потужності ~(ΔV_step/V_mpp)².

2. **Drift при швидких змінах G** — це ключова проблема:
   - Уявімо, що ми збільшили V (ΔV > 0).
   - Тим часом G раптово зросла (промінь сонця між хмарами).
   - P також зросла (ΔP > 0), але **не через нашу дію**.
   - Алгоритм робить хибний висновок «треба ще збільшити V» і **рухається не туди**.
   - Поки алгоритм осягне справжній стан, можна втратити секунди потужності.

3. **Tradeoff швидкість/точність** через ΔV_step:
   - малий крок → точніше у стаціонарі, але повільніше сходження після зміни G;
   - великий крок → швидке сходження, але великі осциляції.
   - **Дослідження цього tradeoff — exp4_sweep_po_step**.

4. **Локальні MPP при частковому затіненні** — за наявності bypass-діодів P-V крива має кілька локальних максимумів. P&O може застрягти в локальному, не знаходячи глобального. (У цій курсовій частково затінення не моделюється — це окрема велика тема.)

5. **Початкова невизначеність** — при холодному старті потрібен час на сходження.

## Що показують експерименти

- **exp1_full_day** — типова добова робота, ефективність ~98-99% при помірних умовах.
- **exp2_transient** — drift при різких змінах G.
- **exp3_sweep_cloudiness** — як ефективність падає зі зростанням мінливості погоди.
- **exp4_sweep_po_step** — оптимальний V_step для конкретного сценарію.
